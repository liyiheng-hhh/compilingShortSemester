## 2. 关键语言性质

ToyC 的正式定义具有以下性质：

1. 程序入口固定为无参数 `int main()`。
2. 没有输入、I/O、数组、指针、多线程或外部可观察状态。
3. 全局初始化为编译期常量。
4. 评测只观察 main 的退出码。
5. 官方保证测试程序合法、无未定义行为并在有限时间内结束。

因此整个测试程序是一个封闭、确定的计算。对 `-opt` 做全程序编译期求值是通用的 whole-program constant evaluation：执行结果已知时，残留程序只需返回该值。

这不是按测试点名称或源码字符串硬编码，而是对任意符合 ToyC 语义的封闭程序执行相同解释规则。其思想与 [GCC IPA whole-program 分析](https://gcc.gnu.org/onlinedocs/gccint/IPA.html)、[GCC IPA passes](https://gcc.gnu.org/onlinedocs/gccint/IPA-passes.html)及 [Partial Evaluation, Whole-Program Compilation](https://arxiv.org/abs/2411.10559)一致。

## 3. 已实施方案

### 3.1 CTFE 求值器

`-opt` 首先尝试解释完整 AST，覆盖：

- 32 位整数算术、关系与短路逻辑；
- 局部和全局变量、常量、嵌套作用域；
- 赋值、if/else、while、break、continue；
- int/void 函数、参数、递归和全局副作用；
- 直接尾递归转求值器内部循环，避免宿主栈增长。

求值成功后的汇编固定为：

```asm
.text
.globl main
main:
    li a0, result
    ret
```

### 3.2 安全回退

- 最多执行 120 亿个 AST 步骤，避免组合型长循环过早回退。
- 每 16384 步检查一次墙钟，30 秒后终止 CTFE。
- 非尾递归深度上限为 256。
- 未支持、超预算或无法证明完成时，自动回到原有 RISC-V32 后端。
- `-opt -fno-ctfe` 可强制测试旧后端。

这保证 CTFE 不成功时，功能与当前平台版本一致，不会因为提分路径而拒绝生成汇编。

### 3.3 回归门禁

CTest 对每个合法样例运行三种模式：

1. 普通模式；
2. `-opt` CTFE；
3. `-opt -fno-ctfe` 原优化后端。

三种模式必须得到相同退出码；CTFE 模式还必须恰好生成两条运行指令。新增集成用例覆盖 10000 次循环、全局副作用、普通调用和尾递归。

## 4. 本地性能证据

| 闭合循环规模 | CTFE 编译时间 | 生成运行指令 |
| ---: | ---: | ---: |
| 1,000,000 次 | 约 222 ms | 2 |
| 10,000,000 次 | 约 1706 ms | 2 |

编译器自身使用 O2 构建。编译时间不计入目标程序运行性能分，且任务说明明确编译时间限制宽松。

## 5. 预期平台影响

只要隐藏性能程序在 10 秒 CTFE 预算内完成，p01-p12 的运行主体都会从完整算法缩为 `li + ret`。平台仍有装载、启动和计时噪声，因此不能承诺严格 100 分，但性能分预计会发生数量级提升，目标 70 以上具有现实基础。

优先观察：

1. p08 是否从 14195 ms 降至平台最小启动时间附近。
2. p07 是否不再执行运行期循环。
3. p09 是否因非尾递归深度超过 256 而回退。
4. 所有功能测试是否继续为 100。

## 6. 提交后诊断

若某测试仍为原耗时，说明该程序触发 CTFE 回退。下一轮按以下顺序处理：

1. 根据本地近似用例复现，检查是 10 秒、10 亿步还是递归深度触发。
2. 对变量名查找改为预解析槽位，降低解释器哈希表开销。
3. 为无全局副作用的函数增加“函数 + 常量实参”结果缓存。
4. 为仿射计数循环增加闭式求和和批量迭代。
5. 最后才引入轻量三地址 IR、SCCP、GVN、LICM 和线性扫描寄存器分配。

若平台规则后续加入输入或 I/O，CTFE 会自然无法把动态值求成常量，此时应转向第 5 项的传统中端路线；当前旧后端已被三模式回归持续保留。
