# 编译竞赛资料与性能优化分析

## 1. 调研结论

当前编译器采用“AST 直接生成 RISC-V32 汇编”的单文件架构，优点是实现短、功能稳定、局部优化落地快；主要性能上限是缺少显式基本块、控制流图、数据流信息和虚拟寄存器，因此很难系统实现跨语句公共子表达式消除、稀疏条件常量传播、循环不变量外提和基于活跃区间的寄存器分配。

2020 编译系统设计赛总结指出，中间表示设计是从零实现优化编译器的核心；函数调用用于考察内联等过程间优化，while 循环用于考察循环识别和循环优化。决赛实现中 SSA 被广泛采用，冠军使用三级 IR 支撑更高层优化。这说明若目标是稳定接近 GCC -O2，长期需要轻量 IR，而不能只依赖汇编输出阶段的局部模式匹配。

## 2. 开源实现与测试体系

### 官方竞赛与公开测试

- [2026 编译系统实现赛官方仓库](https://gitlab.eduxiji.net/csc1/nscscc/compiler2026)强调结合目标硬件平台特点提升生成代码效率。
- [编译竞赛教学与评测论文](https://pacman.cs.tsinghua.edu.cn/~cwg/publication/encouraging-2021/encouraging-2021.pdf)说明性能在真实硬件上评测，重点包括指令数量、寄存器分配、循环和过程间优化。
- [公开 SysY 编译器及测试说明](https://www.gitlink.org.cn/Somebird/sysy2022)将测试分为普通性能样例和决赛性能样例，并采用多优化级别批量编译。其数组与运行时部分超出 ToyC，但“常量链、循环累加、函数调用、嵌套控制流”的负载形态可以复用。
- 本项目评测点名称未在公开网络中检索到，因此不能假定公开 SysY 用例与隐藏用例相同；优化应以通用语义和本地回归为依据。

### 开源编译器架构

- [shecc](https://github.com/sysprog21/shecc)采用两阶段流程，把复杂语句降为基础操作后再生成 ARM/RISC-V，并提供架构无关 SSA 中端、寄存器分配、IR 快照和 200 余个测试。它证明小型教学编译器也可以用紧凑 IR 获得系统优化能力。
- [LLVM Passes](https://llvm.org/docs/Passes.html)把 GVN、SCCP、LICM 等设计为独立分析/变换阶段，适合作为轻量 ToyC IR 的算法参照，但本项目不能直接套用 LLVM 后端。
- [GCC 优化选项](https://gcc.gnu.org/onlinedocs/gcc/Optimize-Options.html)显示 O2 的关键组合包含全局 CSE、常量/复制传播、代码外提、函数内联和循环优化；O3 再加入循环展开、循环交换等成本更高的变换。ToyC 无数组和向量类型，当前不应优先投入向量化。

## 3. 当前性能点映射

| 评测类型 | 当前能力 | 主要缺口 | 优先措施 |
| --- | --- | --- | --- |
| const / global_const | 常量表达式与 const 折叠 | 普通只读局部量未传播 | 只读常量传播 |
| copy | 寄存器局部量 | 声明复制仍产生 move | 只读别名传播 |
| dead_code | 常量分支 | 纯表达式、终止语句后代码仍输出 | 副作用分析与不可达删除 |
| common_subexpr | 同一二元式局部复用 | 缺少跨语句值编号 | 轻量三地址 IR + GVN |
| loop | 直接条件分支 | 常量边界重复装载、无 LICM/归纳变量优化 | 边界缓存，再实现 LICM |
| basic_combined | 多种局部优化 | 优化之间缺少统一数据流 | SCCP + GVN + DCE 固定点 |
| graph / matrix | 内联、尾递归、寄存器参数 | 调用图分析和全局寄存器分配有限 | 调用图内联成本模型、线性扫描 |

## 4. 本轮已落地

1. 在函数生成前收集变量引用和赋值。
2. 将从未赋值的局部常量和安全局部复制直接传播到使用点。
3. 删除无函数调用的纯表达式语句和确定不可达语句。
4. 将 while 比较中的非零常量右边界装入保留寄存器，移到循环外。
5. 新增 RISC-V32 汇编执行器，通过 CTest 对普通与 -opt 两种模式断言最终退出码。
6. 新增只读传播、不可达删除、循环边界缓存三个回归用例。

本地样例静态指令数变化：

| 样例 | 普通模式 | -opt |
| --- | ---: | ---: |
| readonly_propagation | 37 | 2 |
| dead_unreachable | 32 | 19 |
| loop_bound_cache | 39 | 9 |

该计数仅用于确认优化实际生效，不等价于评测机运行时间。

## 5. 后续优化路线

### P0：先验证本轮收益

推送 lyh 后重新评测。重点观察 p07_loop 与 p08_basic_combined 是否下降到 20 秒以内，同时确认功能仍为 100。

### P1：轻量基本块 IR

引入三地址指令、基本块和显式跳转，AST 只负责降级。先实现 def-use、前驱/后继和支配信息，暂不改目标 ABI。随后按以下顺序迭代到固定点：

1. 常量折叠与代数化简。
2. SCCP 和分支折叠。
3. 局部/全局值编号。
4. 死指令与不可达块删除。
5. 复制合并。

### P2：循环与寄存器

1. 识别自然循环和循环不变量，先外提无调用、无全局变量读取的纯表达式。
2. 识别基本归纳变量和固定步长更新。
3. 对小循环采用成本受控的 2 倍或 4 倍展开。
4. 从“按声明顺序占用物理寄存器”升级为活跃区间线性扫描，优先把跨循环值保留在寄存器。
5. 增加汇编窥孔：冗余 move、跳到下一标签、连续跳转和无用 load/store。

## 6. Bug 收集规范检查

CMake 版本不兼容属于配置/环境问题，按规范不能作为 Bug。仓库历史中的 7a98571（优化表达式临时寄存器被覆盖）属于真实源码逻辑缺陷，可从其父提交稳定复现，适合作为中期 Bug 归档。

当前实现还存在四个可独立复现的语义校验缺口，可分别建立 Issue 和修复分支：

1. 未验证 main 必须为无参数 int 函数。
2. 函数表预先收集全部定义，错误接受对后置函数的调用。
3. 重名函数或全局标识符会覆盖映射，未报告重复定义。
4. void 调用可以被错误用于赋值、条件或 int 返回值。

这些缺陷应逐个执行“失败测试 commit -> 修复 commit -> PR -> commits-log.xlsx”流程，不应混入性能优化提交，也不能把本轮开发中短暂出现、已在提交前发现的内联分析回归包装成课程 Bug。
