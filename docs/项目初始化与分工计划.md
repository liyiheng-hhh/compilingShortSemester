# ToyC 编译器项目初始化与分工计划

## 1. 项目目标

四人协作实现 ToyC 到 RISC-V32 汇编的编译器。编译器从标准输入读取 ToyC 源码，向标准输出写出汇编代码，并支持评测系统在功能测试和性能测试中的自动构建、运行与评分。

核心交付物：

- 可构建、可运行的 C++20 编译器项目。
- ToyC 词法、语法、语义、代码生成与基础优化实现。
- 面向评测系统的测试样例、回归测试流程和性能测试记录。
- 实践报告，说明设计方案、实现细节、测试结果、优化策略和分工贡献。

## 2. 初始化准备

### 2.1 仓库与构建

- 已初始化 Git 仓库，并关联远程仓库 `https://github.com/liyiheng-hhh/compilingShortSemester.git`。
- 使用 C++20 与 CMake，入口可执行文件暂定为 `toyc`。
- 编译器接口固定为：
  - `stdin`：ToyC 源程序。
  - `stdout`：RISC-V32 汇编。
  - `stderr`：调试日志或错误信息。
  - `-opt`：性能测试优化开关，可逐步接入优化流水线。

### 2.2 推荐模块边界

```text
source
  -> Lexer
  -> Parser
  -> AST
  -> Semantic Analyzer
  -> IR
  -> Optimizer
  -> RISC-V32 Code Generator
  -> assembly
```

建议先完成正确性闭环，再做性能优化：

1. 最小闭环：`int main(){ return 常量表达式; }` 可编译并返回正确退出码。
2. 表达式与变量：支持局部变量、赋值、算术、关系、逻辑短路。
3. 控制流：支持 `if/else`、`while`、`break`、`continue`。
4. 函数：支持参数传递、递归、返回值、void 函数。
5. 全局：支持全局变量、全局常量、常量编译期求值。
6. 优化：常量折叠、死代码删除、局部公共子表达式、简单寄存器分配。

### 2.3 代码规范

- 主分支保持可构建，功能通过后再合并。
- 每个模块提供最小自测样例，修复 bug 时补回归用例。
- 所有评测接口行为只依赖 `stdin/stdout`，不要在正式路径依赖固定文件名。
- 实践报告中的结论必须能对应到代码、测试数据或性能记录。

## 3. 四人分工

### 3.1 分工原则

本项目采用“功能域负责制 + 公共接口共管”的方式，避免单纯按词法、语法、语义、后端分层后造成前期/后期工作量不均。每位成员负责一个相对完整的功能域，包含语法支持、语义规则、IR/后端对接、测试样例和报告素材；公共接口由全员在早期统一约定，减少后续合并冲突。

公共接口共管：

- AST/IR 基础结构、错误处理格式、测试目录规范由四人共同确认。
- CMake、命令行入口、`stdin/stdout` 评测接口由李维护，其他人提交前共同验证。
- RISC-V32 栈帧、寄存器约定、标签命名、全局数据区格式由刘维护，其他人按接口接入。
- 测试和报告不是单人兜底，每个功能负责人必须同步提交本功能样例和报告要点。

### 李：表达式与编译前端基础

职责范围：

- 维护仓库结构、CMake、命令行入口、Token 定义和 Parser 基础框架。
- 负责表达式子系统：字面量、标识符引用、括号、一元运算、算术运算、关系运算、逻辑运算和短路求值的 AST 表达。
- 实现表达式优先级与结合性解析，处理空白、单行注释和多行注释。
- 对接表达式的常量折叠接口和后端求值接口，保证表达式功能可独立测试。

交付边界：

- 能解析并表示所有 ToyC 表达式。
- 提供表达式解析、常量表达式、逻辑短路相关样例。
- 报告中负责“词法语法设计”和“表达式处理”章节素材。

### 张：声明、作用域与语义规则

职责范围：

- 设计符号表、作用域栈和声明生命周期规则。
- 负责全局/局部变量、全局/局部常量、函数形参的声明、屏蔽和使用检查。
- 实现常量编译期求值，检查 `const` 初始化表达式和常量不可赋值规则。
- 实现类型与语义约束检查：函数返回类型、void 函数调用使用限制、声明后使用、函数名唯一等。

交付边界：

- 能完成声明、作用域、常量和函数签名的语义检查。
- 提供变量/常量、嵌套作用域、错误边界和函数声明顺序相关样例。
- 报告中负责“符号表与语义分析”章节素材。

### 刘：函数机制与 RISC-V32 后端基础

职责范围：

- 设计 IR 到 RISC-V32 的后端接口，维护汇编输出结构。
- 负责函数定义、函数调用、参数传递、返回值、递归、栈帧和局部变量存储。
- 负责 `.text`、`.data`、全局变量初始化、全局常量使用和标签生成规范。
- 提供后端调试方法，帮助其他功能域定位生成汇编后的运行错误。

交付边界：

- 能生成可运行的 RISC-V32 函数汇编，覆盖 main、普通函数和递归调用。
- 提供函数调用、参数传递、返回路径、全局对象相关样例。
- 报告中负责“RISC-V32 代码生成”和“运行时栈帧”章节素材。

### 贾：语句控制流、测试体系与轻量优化

职责范围：

- 负责语句子系统：语句块、空语句、表达式语句、赋值、`if/else`、`while`、`break`、`continue`、`return`。
- 对接控制流语义检查和后端跳转生成，维护循环上下文、分支标签和返回路径样例。
- 建立测试目录、基础测试脚本说明和回归样例清单。
- 负责 `-opt` 下的轻量优化组合：常量折叠落地、死代码删除、简单寄存器复用或局部公共子表达式。

交付边界：

- 能覆盖所有语句和控制流的功能闭环。
- 提供控制流、循环跳转、返回语句、优化前后对比相关样例。
- 报告中负责“测试方法”“优化策略”和“性能对比”章节素材。

### 3.2 均衡性说明

| 成员 | 独立功能域 | 技术难点 | 时间分布 | 协作接口 |
| --- | --- | --- | --- | --- |
| 李 | 表达式与前端基础 | 运算符优先级、短路表达式、Parser 稳定性 | 前期较多，中后期负责表达式优化和集成修正 | AST、表达式 IR |
| 张 | 声明与语义规则 | 作用域、常量求值、返回/调用语义 | 前中期持续推进，后期补语义边界测试 | 符号表、语义结果 |
| 刘 | 函数与后端基础 | 栈帧、调用约定、全局数据、递归 | 中后期较多，前期先定接口和最小后端 | IR、汇编输出 |
| 贾 | 语句控制流与测试优化 | 控制流跳转、循环上下文、优化正确性 | 全程持续，前期建测试，中期做控制流，后期做优化 | 语句 IR、测试规范 |

这样划分后，每个人都有一个主线功能域，既能独立推进，又不会把同一个文件或同一个阶段切得过碎。公共模块只保留少量必要维护人，其他人通过接口协作。

## 5. 验收清单

功能验收：

- 可以通过 CMake 在干净环境中构建。
- 编译器从 `stdin` 读入，向 `stdout` 输出 RISC-V32 汇编。
- `main` 返回值与等价 C 程序一致。
- 支持所有 ToyC 文法：声明、函数、语句、表达式、调用、控制流。
- 支持全局变量、全局常量、局部变量、局部常量和嵌套作用域。
- 正确处理逻辑短路、递归、break/continue、void/int 返回规则。

质量验收：

- 每个新功能都有至少一个正向测试样例。
- 每个已修复 bug 都保留一个回归样例。
- 主分支任意时刻至少能构建成功。
- 提交前运行基础样例，避免把明显失败状态推到远程。

报告验收：

- 包含整体架构图或模块流程图。
- 说明词法/语法/语义/代码生成设计。
- 说明 RISC-V32 栈帧、函数调用、全局变量处理。
- 说明优化策略及开启 `-opt` 前后的效果对比。
- 列出成员分工、贡献和关键提交记录。

## 6. 风险与应对

| 风险 | 表现 | 应对 |
| --- | --- | --- |
| 前后端接口频繁变化 | AST 字段或 IR 约定反复调整 | 第 1 周固定 AST/IR 最小字段，后续只增量扩展 |
| 后端调试困难 | 汇编能生成但返回值错误 | 先做小样例，逐步覆盖表达式、跳转、调用和栈帧 |
| 优化影响正确性 | `-opt` 后功能样例失败 | 优化默认可关闭，每个优化单独开关或单独测试 |
| 报告拖延 | 最后两天缺图表和数据 | 贾从第 1 周开始记录测试和性能数据，全员补实现说明 |
| 合并冲突 | 多人同时改核心节点结构 | 李维护模块边界，重要接口变更先在群内确认 |
